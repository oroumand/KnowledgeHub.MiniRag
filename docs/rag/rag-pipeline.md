# آشنایی با RAG Pipeline در پروژه

در این فایل، جریان کامل داده در سیستم RAG پیاده‌سازی‌شده در پروژه توضیح داده می‌شود.

هدف این توضیح این است که دقیقاً بدانیم:
- داده چطور وارد سیستم می‌شود
- چطور پردازش می‌شود
- چطور جستجو انجام می‌شود
- و چطور پاسخ نهایی تولید می‌شود

---

# RAG چیست؟

RAG مخفف:

Retrieval-Augmented Generation

یعنی:
> ترکیب جستجوی معنایی با مدل زبانی

به زبان ساده:
- اول اطلاعات مرتبط پیدا می‌شود
- بعد مدل فقط بر اساس آن اطلاعات پاسخ می‌دهد

---

# نمای کلی Pipeline

سیستم ما سه مرحله اصلی دارد:

1. Ingestion (ورود داده)
2. Retrieval (جستجوی معنایی)
3. Generation (تولید پاسخ)

---

# 1. Ingestion (ثبت و پردازش داده)

این مرحله زمانی اجرا می‌شود که یک سند جدید وارد سیستم شود.

## مراحل:

### 1. دریافت سند
کاربر از طریق API یک متن ارسال می‌کند:

POST /documents

---

### 2. Chunk کردن متن

متن به بخش‌های کوچک‌تر (chunk) تقسیم می‌شود.

چرا؟
- مدل‌ها محدودیت توکن دارند
- جستجوی دقیق‌تر می‌شود

---

### 3. ذخیره در SQL Server

اطلاعات زیر ذخیره می‌شود:

- SourceDocument
- DocumentChunk

SQL Server منبع اصلی داده‌ها است.

---

### 4. تولید Embedding

برای هر chunk:

متن → embedding (بردار عددی)

---

### 5. ذخیره در Qdrant

هر chunk به صورت یک vector در Qdrant ذخیره می‌شود.

اطلاعات ذخیره‌شده:

- vector
- payload (DocumentId, ChunkId, ...)
- VectorRecordId

---

# نتیجه Ingestion

در پایان این مرحله:

- متن کامل در SQL ذخیره شده
- chunkها در SQL ذخیره شده
- embeddingها در Qdrant index شده‌اند

---

# 2. Retrieval (جستجوی معنایی)

این مرحله هنگام جستجوی کاربر اجرا می‌شود.

## مراحل:

### 1. دریافت query

POST /search/semantic

---

### 2. تولید embedding از query

سوال کاربر → embedding

---

### 3. جستجو در Qdrant

Qdrant نزدیک‌ترین vectorها را پیدا می‌کند.

خروجی:
- ChunkId
- Score (میزان شباهت)

---

### 4. خواندن داده از SQL

با استفاده از ChunkId:

- متن chunk
- عنوان سند
- اطلاعات کامل

از SQL خوانده می‌شود

---

# نتیجه Retrieval

یک لیست از chunkهای مرتبط با سوال کاربر داریم.

---

# 3. Generation (تولید پاسخ)

این مرحله در endpoint چت اجرا می‌شود:

POST /chat/ask

---

## مراحل:

### 1. اجرای semantic search

ابتدا همان retrieval انجام می‌شود

---

### 2. ساخت prompt

یک prompt ساخته می‌شود شامل:

- context (chunkها)
- سوال کاربر

---

### 3. ارسال به مدل Chat

prompt → OpenAI Chat Model

---

### 4. تولید پاسخ

مدل:
- فقط بر اساس context پاسخ می‌دهد
- اگر جواب نداشت → اعلام می‌کند

---

### 5. بازگشت پاسخ + منابع

خروجی شامل:

- answer
- sources (chunkها)

---

# نقش هر بخش در سیستم

## SQL Server

- نگهداری داده اصلی
- متن کامل و chunkها

---

## Qdrant

- جستجوی سریع معنایی
- نگهداری embeddingها

---

## OpenAI

- تولید embedding
- تولید پاسخ

---

# چرا این معماری؟

این طراحی چند مزیت مهم دارد:

- separation of concerns
- مقیاس‌پذیری بهتر
- امکان re-index
- دیباگ راحت‌تر

---

# نکات مهم

## 1. SQL منبع حقیقت است

Qdrant فقط index است

---

## 2. Vector DB فقط retrieval است

متن واقعی همیشه از SQL خوانده می‌شود

---

## 3. Prompt ساده نگه داشته شده

برای نسخه 1:
- هدف clarity است نه پیچیدگی

---

## 4. Chat stateful نیست

هر درخواست مستقل است

---

# محدودیت‌های نسخه 1

- chat history ندارد
- retry ندارد
- background processing ندارد
- indexing status ندارد

---

# جمع‌بندی

در این پروژه:

- داده وارد می‌شود
- chunk می‌شود
- embedding تولید می‌شود
- در vector DB index می‌شود
- semantic search انجام می‌شود
- و در نهایت پاسخ grounded تولید می‌شود

این دقیقاً همان چیزی است که یک سیستم RAG پایه انجام می‌دهد.
